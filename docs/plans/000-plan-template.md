## 작업 목표

## 구현 범위
- 이번 Plan에서 실제로 구현하거나 변경할 항목을 적는다.
- 파일, 기능, API, UI 단위로 구체화한다.

## 제외 범위
- 이번 Plan에서 의도적으로 하지 않을 일을 적는다.
- 후속 Plan으로 넘길 항목도 여기에 적는다.

## 성공 조건
- 구현이 완료되었다고 판단할 수 있는 조건을 적는다.
- 사용자 관점의 동작과 개발 관점의 구조 조건을 함께 적는다.

## UI/UX 흐름
Frontend 화면 작업인 경우 작성한다.

- 사용자가 어떤 화면에서 시작하는지 적는다.
- 어떤 질문 또는 단계로 입력을 받는지 적는다.
- 결과가 어떤 리포트 형태로 표시되는지 적는다.
- 빈 상태, 로딩, 실패, 성공 상태를 적는다.

## Mock 데이터 기준
Mock UI 작업인 경우 작성한다.

- Mock 데이터가 실제 계산 로직을 대체하지 않는다는 점을 명시한다.
- 향후 API 응답으로 교체 가능한 타입 구조를 적는다.
- Mock 데이터 위치와 제거 또는 교체 기준을 적는다.

## 테스트 조건
Frontend 작업인 경우:
- `npm run lint`
- `npm run type-check`
- `npm run test`
- `npm run build`

Backend 작업인 경우:
- `./gradlew test`
- `./gradlew build`

Infra 작업인 경우:
- 필요한 Docker 또는 배포 검증 명령을 적는다.

해당하지 않는 검증은 생략 사유를 남긴다.
